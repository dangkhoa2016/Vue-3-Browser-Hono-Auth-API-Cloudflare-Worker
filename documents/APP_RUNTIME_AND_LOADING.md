This document explains how the browser-only runtime starts, how Vue SFC modules are loaded, and which parts are risky to change.

## 1. Main Files

- `assets/js/init.js`
- `assets/js/router.js`
- `vue/App.vue`
- `vue/Loader.vue`
- `vue/components/AsyncLoader.vue`
- `assets/js/appServices.js`

## 2. Boot Sequence

1. `assets/js/init.js` creates the shared SFC loader options object.
2. It registers module cache entries for `vue`, `vue-i18n`, and app service helpers before route loading starts.
3. It stores those options through `setSfcOptions()` so other modules can read them.
4. It loads the initial language before the loader UI mounts.
5. It mounts `vue/Loader.vue` into `#loader`.
6. `window.initMainApp` is exposed so the loader can trigger the full app mount.
7. During main app init, Pinia, router, i18n, and store singletons are wired together.

## 3. Why The Order Matters

`assets/js/router.js` uses top-level `await` to preload `AsyncLoader.vue`. Because of that, `init.js` must populate SFC loader cache entries before dynamically importing the router module.

If this order changes, the app can fail early with missing module or duplicated singleton problems.

## 4. Runtime Loading Model

### Page loading

- Every route points to a wrapper component generated by `load(path)` in `router.js`.
- The wrapper renders `AsyncLoader.vue`.
- `AsyncLoader.vue` lazy-loads the real page SFC and handles retries or error display.

### Cache behavior

- Runtime module cache is handled by `vue3-sfc-loader`.
- Browser Cache Storage is used in `init.js` through `SFC_CACHE_NAME`.
- Cache is bypassed in localhost mode and when `__reload` exists in the URL.

### App-level keep-alive

- `vue/App.vue` builds a `keepAliveRouteNames` list from route metadata.
- Only routes with `meta.keepAlive` are cached.
- This avoids re-fetching some expensive admin pages while leaving the rest uncached.

## 5. Shared Singleton Model

The app avoids ad-hoc globals by registering shared instances through `assets/js/appServices.js`.

Important singletons:

- SFC loader options
- Auth store instance
- Modal store instance
- Toast store instance

This pattern reduces direct `window.*` coupling, but it still depends on correct initialization timing.

## 6. Review Risks

### High-risk changes

- Reordering imports or cache registration inside `init.js`
- Changing router module load timing
- Replacing `createWebHashHistory()` without checking static hosting constraints
- Removing `keepAlive` metadata without checking operator workflows
- Introducing feature logic into `init.js` or `router.js`

### Typical failure symptoms

- Loader mounts but the main app never appears
- Pages render with duplicated stores or missing injected services
- Async route loading fails only in production cache mode
- Previously cached admin screens start losing state between navigations

## 7. Optimization Ideas

1. Extract cache and asset handling from `init.js` into a dedicated runtime loader helper.
2. Add a small runtime diagnostic mode for module cache and loader failures.
3. Keep `router.js` thin by moving auth guard creation into a dedicated function.
4. Document route metadata conventions next to route definitions if more route types are added.

## 8. Safe Change Checklist

1. Loader still mounts before the main app.
2. Initial language still loads before UI text is shown.
3. Router import still happens after required module cache entries exist.
4. Toast, auth, and modal stores remain singletons.
5. Protected routes still work with direct URL entry.
